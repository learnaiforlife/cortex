# User-Level File Formats

Specification for user-level files generated by Cortex Discover. These files live in `~/.claude/` and apply across all projects.

---

## 1. ~/.claude/CLAUDE.md Format

The user-level CLAUDE.md contains developer identity, global preferences, and integration context. It must NOT duplicate project-level content.

### What belongs in user-level CLAUDE.md

- **Developer identity**: Name, role, preferred languages, experience level
- **Global preferences**: Code style preferences that apply everywhere (indentation, naming conventions, etc.)
- **Integrations summary**: Which integrations are configured and how to use them
- **Company context**: Organization name, team structure, shared conventions
- **Tool preferences**: Preferred test frameworks, linters, formatters across projects
- **Personal workflow notes**: How the developer prefers to work with Claude Code

### What belongs in project-level CLAUDE.md (NOT user-level)

- Architecture and project structure
- Build/test/lint/deploy commands
- Project-specific conventions and patterns
- API documentation and endpoints
- Database schema information
- CI/CD pipeline details
- Dependencies and version requirements

### NEVER duplicate between levels

If something is in user-level CLAUDE.md, do NOT repeat it in project-level CLAUDE.md. The two files are merged at runtime with project-level taking precedence.

### Example user-level CLAUDE.md

```markdown
<!-- Generated by Cortex Discover -- last updated: 2025-01-15T10:30:00Z -->

# Global Claude Code Context

## User Info
- Senior full-stack engineer at Acme Corp
- Primary languages: TypeScript, Python, Go
- Prefers functional patterns over OOP where practical

## Code Style Preferences
- Use 2-space indentation for JS/TS, 4-space for Python
- Prefer named exports over default exports
- Always use strict TypeScript (no `any`)

## Configured Integrations
- **Jira**: MCP server configured. Use for issue tracking (ACME project).
- **Slack**: MCP server configured. Use for team notifications.
- **GitHub**: MCP server configured. Use for PR and issue workflows.
- **Sentry**: MCP server configured. Use for error investigation.

## Workflow Preferences
- Always run tests before suggesting a commit
- Prefer small, focused commits over large batches
- Use conventional commit messages (feat:, fix:, chore:, etc.)
```

## 2. ~/.claude/.mcp.json Format

User-level MCP server configuration. Only integration servers belong here -- never project-specific servers.

### Rules

- **Include**: Integration servers (Jira, Slack, GitHub, Linear, Notion, Confluence, Sentry, Datadog, PagerDuty, GitLab)
- **NEVER include**: Project-specific servers (postgres, sqlite, filesystem, playwright, puppeteer)
- **Credential format**: Always use `${ENV_VAR}` syntax -- never hardcode secrets
- **Structure**: Identical to project-level `.mcp.json`

### Example ~/.claude/.mcp.json

```json
{
  "mcpServers": {
    "jira": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@anthropic-ai/mcp-server-jira"],
      "env": {
        "JIRA_URL": "${JIRA_URL}",
        "JIRA_EMAIL": "${JIRA_EMAIL}",
        "JIRA_API_TOKEN": "${JIRA_API_TOKEN}"
      }
    },
    "slack": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@anthropic-ai/mcp-server-slack"],
      "env": {
        "SLACK_BOT_TOKEN": "${SLACK_BOT_TOKEN}",
        "SLACK_TEAM_ID": "${SLACK_TEAM_ID}"
      }
    },
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "sentry": {
      "type": "http",
      "url": "https://mcp.sentry.dev/mcp",
      "env": {
        "SENTRY_AUTH_TOKEN": "${SENTRY_AUTH_TOKEN}"
      }
    }
  }
}
```

### What NOT to put in user-level .mcp.json

```json
// WRONG -- these are project-specific and must stay in project-level .mcp.json
{
  "mcpServers": {
    "postgres": { "..." },
    "sqlite": { "..." },
    "filesystem": { "..." },
    "playwright": { "..." }
  }
}
```

## 3. ~/.claude/rules/*.md Format

Global rules that apply to all projects. Same format as project-level rules but without path restrictions.

### Rules

- Same format as project-level rules (`.claude/rules/*.md`)
- Optional YAML frontmatter with `alwaysApply` field
- Should NOT have `paths:` restriction -- they apply to all files in all projects
- Keep concise -- max 30 lines per rule file
- Filename should be descriptive (e.g., `commit-style.md`, `code-review.md`)

### Example: ~/.claude/rules/commit-style.md

```markdown
---
alwaysApply: true
---

# Commit Message Convention

- Use conventional commits: feat:, fix:, chore:, docs:, refactor:, test:
- Keep subject line under 72 characters
- Use imperative mood ("add feature" not "added feature")
- Include ticket reference when available (e.g., JIRA-1234)
```

### Example: ~/.claude/rules/security.md

```markdown
---
alwaysApply: true
---

# Security Rules

- Never commit secrets, tokens, or API keys
- Always use environment variables for credentials
- Never log sensitive data (passwords, tokens, PII)
- Prefer parameterized queries over string concatenation for SQL
```

### What NOT to put in global rules

- Project-specific linting configurations
- Framework-specific conventions (React, Django, etc.)
- Path-specific rules (those belong in project-level rules with `paths:` frontmatter)

## 4. Merge Precedence

When user-level and project-level files conflict, the following precedence applies:

### CLAUDE.md

- Project-level CLAUDE.md **overrides** user-level CLAUDE.md for that project
- Both are loaded, but if they contain contradictory instructions, project-level wins
- Example: User-level says "use 2-space indent" but project-level says "use 4-space indent" -- project-level wins for that project

### .mcp.json

- Project-level and user-level MCP servers **merge additively**
- If both levels define the same server name, project-level configuration wins
- Example: User-level has `github` server; project-level also has `github` server with different token -- project-level `github` is used
- Servers unique to each level are all available (user-level `jira` + project-level `postgres` = both available)

### rules/*.md

- Project-level rules **override** user-level rules with the **same filename**
- Example: Both have `security.md` -- project-level `security.md` takes precedence
- Rules with different filenames from each level are all active
- Example: User-level `commit-style.md` + project-level `testing.md` = both active

### Precedence summary table

| File | Conflict Resolution |
|------|-------------------|
| `CLAUDE.md` | Project-level overrides user-level |
| `.mcp.json` | Additive merge; project-level wins on name collision |
| `rules/*.md` | Project-level overrides same-name user-level rules |

## 5. File Header Convention

All files generated by Cortex Discover must start with the following header:

```
<!-- Generated by Cortex Discover -- last updated: {ISO timestamp} -->
```

### Format details

- Use ISO 8601 timestamp format: `YYYY-MM-DDTHH:MM:SSZ` (UTC)
- Use `--` (double dash) not `---` inside HTML comments to avoid markdown parsing issues
- The header must be the very first line of the file
- Leave one blank line between the header and the first content

### Examples

For markdown files (CLAUDE.md, rules/*.md):
```markdown
<!-- Generated by Cortex Discover -- last updated: 2025-01-15T10:30:00Z -->

# File Title
...
```

For JSON files (.mcp.json):
```
// Note: JSON does not support comments. For .mcp.json, the header is omitted.
// Instead, add a "_generated" key:
{
  "_generated": "Cortex Discover -- last updated: 2025-01-15T10:30:00Z",
  "mcpServers": { ... }
}
```

### Updating the timestamp

When Cortex Discover updates an existing generated file, it must:
1. Check for the existing header
2. Update the timestamp to the current time
3. Preserve all user modifications below the header
